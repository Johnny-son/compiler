# 架构设计

## 1. 总体流水线

项目采用三段式编译流程：

```text
frontend -> ir -> backend
```

数据流如下：

```text
source(.c)
  -> ANTLR 词法/语法分析
  -> CSTVisitor 生成 AST
  -> IRGenerator 生成线性 IR
  -> BackendDriver 串联后端各模块生成汇编(.s)
```

## 2. 主流程入口

主控逻辑在：`src/main.cpp`

核心阶段：

1. 参数解析（`ArgsAnalysis`）
2. 前端生成 AST（`ASTGenerator`）
3. 中端生成 IR（`IRGenerator` + `Module`）
4. 后端生成 RISCV64 汇编（`BackendDriver`）

## 3. Frontend

关键文件：

- `src/frontend/antlr4/MiniC.g4`：语法定义
- `src/frontend/ASTGenerator.cpp`：ANTLR 入口，负责得到 CST
- `src/frontend/CSTVisitor.cpp`：将 CST 转为 AST
- `src/frontend/include/AST.h`：AST 节点结构定义

## 4. IR（中间表示）

关键文件：

- `src/ir/IRGenerator.cpp`：AST 到 IR 的核心翻译
- `src/ir/include/Module.h`：模块级上下文（函数、全局变量、作用域）
- `src/symboltable/ScopeStack.*`：作用域栈
- `src/ir/include/Instruction.h`：IR 指令基类

实现方式：

- 通过 AST 节点类型到处理函数的映射表分发翻译逻辑。
- 每个 AST 节点会产出对应的 `Value` 与 `InterCode` 片段。

## 5. Backend（RISCV64）

关键文件：

- `src/backend/include/BackendDriver.h`、`src/backend/BackendDriver.cpp`：后端总入口，负责调度各阶段
- `src/backend/include/IRAdapter.h`、`src/backend/IRAdapter.cpp`：IR 适配层，向后端提供统一的只读视图
- `src/backend/include/FrameLayout.h`、`src/backend/FrameLayout.cpp`：栈帧与对象布局
- `src/backend/include/InstructionSelector.h`、`src/backend/InstructionSelector.cpp`：IR 到 RV64 指令模板翻译
- `src/backend/include/Asm.h`、`src/backend/Asm.cpp`：汇编指令对象与最终文本输出

当前后端采用的最小链路为：

```text
Module
  -> IRAdapter
  -> FrameLayoutBuilder
  -> InstructionSelector
  -> AsmPrinter
```

设计说明：

- `IRAdapter` 不重新构造一套 IR，只对现有 `Module / Function / Instruction / Value` 提供后端友好的只读包装。
- `FrameLayoutBuilder` 为形参、局部变量、IR 临时结果分配栈槽，统一采用 8 字节槽位，栈帧总大小按 16 字节对齐。
- `InstructionSelector` 目前内置一套最小寄存器使用策略，固定使用少量工作寄存器（如 `t0/t1/t2/t3`），不单独实现复杂寄存器分配。
- `AsmPrinter` 只负责把结构化汇编对象打印成合法 `.s` 文本，不承担语义翻译职责。

当前已接通的指令模板包括：

- `entry / exit`
- `assign`
- `add / sub / mul / div / mod`
- `call`
- `label / goto`

## 6. 当前状态与扩展方向

当前框架已支持从 MiniC 子集到 IR/ASM 的完整链路，并具备 CI 测试入口。

可优先扩展方向：

1. 继续补全全局变量、函数调用、控制流等后端细节并扩大通过用例范围
2. 将当前写死在指令选择中的寄存器使用策略逐步抽离成独立的寄存器分配模块
3. 在保持接口稳定的前提下继续细化汇编对象层与后端驱动流程
