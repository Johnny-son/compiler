# Frontend 架构详解

本文档只聚焦 `src/frontend`，并按“总览 -> 数据流 -> 每个文件职责”展开。

二层超细化入口：

- [Frontend 二层超细化版](./Frontend二层超细化版.md)

## 1. 模块定位

`frontend` 负责把源代码文本变成抽象语法树 AST，并提供 AST 图形化输出能力。

前端不直接产生汇编，也不负责寄存器、栈帧等后端问题。它的核心交付物是：

- 语义结构化的 `ast_node *` 根节点
- 可选的 AST 图片（Graphviz）

## 2. 前端主流程（代码级）

入口链路如下：

1. `ASTGenerator::run()` 打开输入文件。
2. `MiniCLexer` 做词法切分，`MiniCParser` 做语法分析。
3. `MiniCErrorListener` 捕获词法/语法错误并通过 `Status::Error` 返回。
4. `MiniCCSTVisitor::run()` 深度遍历 CST，构建项目自定义 AST。
5. 如果命令行指定 `-T`，`OutputAST()` 把 AST 导出为图片。

## 3. 关键数据结构

前端统一使用 `ast_node` 表示抽象语法树节点，核心字段有：

- `node_type`：节点类别（例如 `AST_OP_FUNC_DEF`、`AST_OP_ADD`）
- `sons`：孩子节点列表
- `type/name/integer_val/float_val/line_no`：语义属性
- `needScope`：供中端控制是否在 block 上自动进出作用域
- `blockInsts`、`val`：中端 IR 生成过程复用字段

这意味着 AST 结构不仅承载语法信息，也预留了中端翻译时的暂存位。

## 4. 文件级详解

### 4.1 语法定义文件

#### `src/frontend/antlr4/MiniC.g4`

主要作用：

- 定义 MiniC 的词法规则与语法规则。
- 是自动生成 `MiniCLexer/MiniCParser/...` 的唯一输入源。

当前文法覆盖：

- 编译单元：`compUnit: (funcDef | varDecl)* EOF;`
- 函数定义：`int` 返回、可选 `int` 标量形参列表、函数体
- 声明：`int` 变量声明、逗号分隔、可选初始化
- 语句：`return`、赋值、块、表达式语句、空语句
- 表达式：`addExp/mulExp/unaryExp/primaryExp`
- 函数调用：`ID '(' realParamList? ')'`
- 词法 token：关键字、标识符、数字、注释、空白

与其它文件关系：

- 被 CMake 调用 ANTLR4 生成 `src/frontend/autogenerated/*`。
- `ASTGenerator.cpp` 依赖生成的 lexer/parser 类。

### 4.2 AST 基础类型与工厂

#### `src/frontend/include/AttrType.h`

主要作用：

- 定义词法属性与语义属性的轻量结构。

核心定义：

- `enum class BasicType`：`TYPE_INT/TYPE_VOID/TYPE_FLOAT/...`
- `digit_int_attr`：整数值 + 行号
- `digit_real_attr`：浮点值 + 行号
- `var_id_attr`：标识符指针 + 行号
- `type_attr`：类型 + 行号

典型使用点：

- `CSTVisitor.cpp` 在访问 token 时构造 `type_attr`、`var_id_attr`。
- `AST.cpp` 用这些属性创建 `ast_node`。

#### `src/frontend/include/AST.h`

主要作用：

- 定义 AST 节点类型枚举与 `ast_node` 类接口。

核心内容：

- `enum class ast_operator_type`：叶子节点 + 内部节点（函数定义、块、return、赋值、二元运算等）。
- `ast_node` 构造/工厂接口：`New(...)`、`create_func_def(...)`、`create_func_call(...)` 等。
- 树操作接口：`insert_son_node`、`Delete`。
- 中端复用字段：`blockInsts`、`val`。

该头文件是整个前端和中端沟通的核心数据协议。

#### `src/frontend/AST.cpp`

主要作用：

- 实现 `ast_node` 全部构造逻辑、工厂逻辑、内存回收逻辑。

关键实现点：

- 多种叶子节点构造（整数字面量、标识符、类型节点）。
- `isLeafNode()` 决定图形化与遍历策略。
- `insert_son_node()` 负责父子关系建立。
- `Delete()` 递归释放 AST，防止泄漏。
- `create_func_def/create_func_call/create_var_decl_stmt_node` 等工厂简化访问器代码。
- `typeAttr2Type()` 把 `BasicType` 转成 IR 层类型对象（如 `IntegerType::getTypeInt()`）。

与中端关系：

- AST 节点的 `type` 字段和 `needScope` 直接影响 `IRGenerator` 行为。

### 4.3 CST 到 AST 生成器

#### `src/frontend/include/ASTGenerator.h`

主要作用：

- 定义前端总入口类 `ASTGenerator`。

公开接口：

- 构造函数：传入源码路径。
- `run()`：执行解析并构建 AST。
- `getASTRoot()`：取回 AST 根节点。

#### `src/frontend/ASTGenerator.cpp`

主要作用：

- 串起 ANTLR 输入流、词法器、语法器和 CST visitor。

关键实现点：

- 使用 `antlr4::ANTLRInputStream` 读取文件。
- 构造 `MiniCLexer` 与 `MiniCParser`。
- 注入 `MiniCErrorListener`，统一语法错误输出。
- 从 `parser.compUnit()` 获取 CST 根。
- 调用 `MiniCCSTVisitor::run()` 产出 AST 根。

错误处理策略：

- 文件打不开、语法分析失败、`cstRoot` 无效都会返回 `false`。

#### `src/frontend/include/CSTVisitor.h`

主要作用：

- 声明 `MiniCCSTVisitor`，继承 `MiniCBaseVisitor`。

公开能力：

- `run(CompUnitContext*)`：返回 `ast_node *`。
- 覆盖大量 `visitXxx`：按语法规则逐一生成 AST。

#### `src/frontend/CSTVisitor.cpp`

主要作用：

- 真正把 ANTLR 的 CST 节点转换为项目定义 AST。

关键实现点（按语义类型）：

- 编译单元：`visitCompUnit` 先处理全局声明，再处理函数。
- 函数定义：`visitFuncDef` 组装返回类型、函数名、函数体。
- 块与语句：`visitBlock/visitBlockItemList/visitStatement`。
- 表达式树：`visitAddExp/visitMulExp/visitUnaryExp/visitPrimaryExp` 维护结合性。
- 调用与参数：`visitRealParamList` 构建 `AST_OP_FUNC_REAL_PARAMS`。
- 变量声明：`visitVarDecl` 支持逗号声明与可选初始化表达式。

重要设计细节：

- 一元负号被翻译成 `0 - expr` 的二元减法树，复用后续中端算术逻辑。
- 空语句返回 `nullptr`，由上层 block 插入时跳过。

### 4.4 AST 图形化

#### `src/frontend/include/Graph.h`

主要作用：

- 声明 `OutputAST(ast_node * root, std::string filePath)`。

#### `src/frontend/Graph.cpp`

主要作用：

- 使用 Graphviz C API 把 AST 渲染成图。

关键实现点：

- `getNodeName()`：把 AST 节点映射为可读标签。
- `genLeafGraphNode()`：叶子节点样式（方框、填充色）。
- `genInternalGraphNode()`：内部节点样式与父子边连接。
- `graph_visit_ast_node()`：递归 DFS 生成图节点。
- `OutputAST()`：创建图上下文、布局、按后缀选择输出格式。

编译开关：

- 仅在 `USE_GRAPHVIZ` 宏下启用；否则 `OutputAST` 空实现。

### 4.5 ANTLR 自动生成文件（`src/frontend/autogenerated/`）

这一组文件由 CMake 在构建期自动生成，原则上不手改。

#### `src/frontend/autogenerated/MiniCLexer.h`

主要作用：

- 词法分析器类声明。
- 定义 token 类型、词法规则入口接口。

#### `src/frontend/autogenerated/MiniCLexer.cpp`

主要作用：

- 词法规则执行实现。
- 把字符流切分为 token 流供 parser 消费。

#### `src/frontend/autogenerated/MiniCParser.h`

主要作用：

- 语法分析器声明。
- 为 `compUnit/funcDef/expr/...` 等规则生成 context 类型。

#### `src/frontend/autogenerated/MiniCParser.cpp`

主要作用：

- 递归下降/预测分析实现。
- 按 `MiniC.g4` 规则构建 CST。

#### `src/frontend/autogenerated/MiniCVisitor.h`

主要作用：

- 访问者接口声明（每个 grammar rule 一个访问入口）。

#### `src/frontend/autogenerated/MiniCVisitor.cpp`

主要作用：

- 访问者的默认实现框架。

#### `src/frontend/autogenerated/MiniCBaseVisitor.h`

主要作用：

- BaseVisitor 声明，默认遍历行为定义在基类里。
- 项目中的 `MiniCCSTVisitor` 继承它并覆盖需要的规则。

#### `src/frontend/autogenerated/MiniCBaseVisitor.cpp`

主要作用：

- BaseVisitor 默认行为实现。

#### `src/frontend/autogenerated/MiniC.tokens`

主要作用：

- 记录 grammar token 编号与符号名映射。
- 供 parser/调试工具使用。

#### `src/frontend/autogenerated/MiniCLexer.tokens`

主要作用：

- 词法器层面的 token 映射输出。

#### `src/frontend/autogenerated/MiniC.interp`

主要作用：

- parser 解释表数据（ATN/状态机序列化结果）。

#### `src/frontend/autogenerated/MiniCLexer.interp`

主要作用：

- lexer 解释表数据。

## 5. 设计边界与扩展建议

当前前端边界：

- 语法仍是 MiniC 子集，不含完整 C 特性。
- 某些语义规则仅做结构转换，尚未在前端阶段做强校验。

建议扩展顺序：

1. 在 `MiniC.g4` 增量扩语法（条件、循环、比较、逻辑表达式）。
2. 在 `CSTVisitor.cpp` 同步补 AST 节点与构造规则。
3. 在 `AST.h/AST.cpp` 扩 `AST_OP_*` 与工厂函数。
4. 用 AST 图导出先验收结构正确，再推进中端翻译。
