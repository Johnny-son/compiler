# 诊断测试框架

## 1. 概述

诊断测试（Error Regression Test）用于验证编译器的错误检测能力。通过在测试用例中嵌入元数据注释，声明**期望的错误码**、**行号**和**关键符号**，脚本自动比对编译器的实际错误输出。

## 2. 元数据格式

在 `.c` 文件**前 10 行**内嵌入 Expected Error Tags：

```c
// @expected-error E1400 x at line 4
int main() {
    int x = 1;
    int x = 2;
    return x;
}
```

### 2.1 语法规则

```bnf
@expected-error-tag ::= '//' @expected-error <error-code> <key-symbol> at line <line-number>
                      | '/*' @expected-error <error-code> <key-symbol> at line <line-number> */

error-code        ::= 'E' [0-9]{4}
key-symbol        ::= 符号名（变量名/函数名等）
line-number       ::= 正整数
```

### 2.2 字段含义

| 字段 | 说明 | 匹配方式 |
|------|------|----------|
| **Error Code** (E####) | 错误定性，必须精确匹配 | `grep -q "E${code}"` |
| **Line Number** | 错误发生位置，用于定位 | 精确匹配（-1 表示"未知行"） |
| **Key Symbol** | 错误涉及的对象名 | substring 包含匹配 |

**多错误支持**：一个文件可有多行 `@expected-error`，全部匹配才算通过。

```c
// @expected-error E1102 foo at line 6
// @expected-error E1200 bar at line 10
int foo(int a) { return a; }
int foo(int b) { return b + 1; }

int bar() { return not_exist(); }
```

## 3. 错误输出格式

所有 IR 错误统一格式为：

```
IR错误[E####] [第N行|未知行] 分类: 详情...
```

示例：

```
IR错误[E1400] 未知行 符号检查: 变量(x)已经存在
IR错误[E1010] 第3行 符号检查: 标识符(not_declared)未定义
IR错误[E1200] 第3行 语义检查: 函数(no_such_func)未定义或未声明
IR错误[E1102] 第1行 符号检查: 函数(main)重复定义或符号冲突
IR错误[E1201] 第8行 参数检查: 函数(id)调用参数个数不匹配，期望1个，实际2个
IR错误[E1301] 第2行 全���初始化: 全局变量(h)初始化目前只支持常量整数表达式
```

## 4. 匹配算法

对于每个 `@expected-error E#### KEY at line N`，脚本执行三维度匹配：

1. **E码匹配**：stderr 中存在 `IR错误[E####]`
2. **Key Symbol 匹配**：该行详情部分包含 `KEY`（substring 匹配）
3. **行号匹配**：
   - 若期望 `at line N` 且 N>0，actual 输出必须是 `第N行`
   - 若期望 `at line -1`，actual 输出必须是 `未知行`

全部条件满足视为该错误**匹配成功**。

## 5. 测试运行

### 5.1 独立运行

```bash
# 运行单个诊断测试
bash scripts/run_ci_tests.sh -err test/stage1_params/param_redeclare_local.c

# 运行多个诊断测试
bash scripts/run_ci_tests.sh -err test/stage1_params/param_redeclare_local.c \
                              test/stage1_params/param_undef_var.c \
                              test/stage1_params/param_call_undef.c

# 运行整个目录
bash scripts/run_ci_tests.sh -err test/stage1_params
```

### 5.2 通过 plan 文件运行

```bash
# 运行 plan 中所有 err 测试
bash scripts/run_ci_tests.sh --all --compiler ./build/compiler

# 只运行 err section
bash scripts/run_ci_tests.sh -err --all
```

### 5.3 环境变量

| 变量 | 说明 | 默认值 |
|------|------|--------|
| `COMPILER_BIN` | 编译器二进制路径 | 自动查找 build/compiler |
| `MINIC_EXTRA_ARGS` | 额外编译器参数 | - |
| `NO_COLOR` | 禁用彩色输出 | - |

## 6. 新增测试用例

### 6.1 步骤

1. 在 `test/stage1_params/` 或其他测试目录下创建 `.c` 文件
2. 在前 10 行添加 `@expected-error` 注释
3. 确认测试用例确实会产生对应错误
4. 在 `scripts/ci_test_plan.txt` 的 `err:` section 添加该用例
5. 运行 `bash scripts/run_ci_tests.sh -err <case>` 验证

### 6.2 示例

```c
// @expected-error E1200 missing_func at line 3
int main() {
    return missing_func();
}
```

## 7. 设计原则

- **错误码稳定**：E码按功能分段（见错误码表），不应频繁改动
- **Key Symbol 精确**：取实际报错中的变量/函数名，便于定位
- **行号敏感**：错误位置变化时测试失败，帮助发现回归
- **无外部依赖**：期望值写在源码注释中，不依赖外部 JSON
- **向后兼容**：未来修改错误消息文本（如 "redefined" → "is already defined"），只要 E码和 Key Symbol 不变，测试仍能通过
